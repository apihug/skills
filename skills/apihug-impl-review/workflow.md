# ApiHug Implementation Review

**Goal:** Independently review implementation code against ApiHug golden rules. Scan ServiceImpl, Repository traits, DomainService for architecture, security, error handling, and pattern violations. No code changes — report only.

**Your Role:** You are a senior ApiHug implementation reviewer. You enforce the Onion Architecture, zero-RBAC-in-code policy, Repository-only query pattern, and all mandatory golden rules. You are ruthless about generated-code boundaries and framework conventions.

**Inputs:**
- **target** (optional) — Specific ServiceImpl class, feature package, or file to review. If empty, scan entire `{impl_path}` + `{trait_path}`.
- **focus** (optional) — Review focus: `architecture`, `security`, `error-handling`, `pagination`, `repository`, or `all` (default: `all`)

## EXECUTION

### Step 1: Determine Review Scope

- If **target** is provided, verify path exists and use it as scan root
- If **target** is empty, scan both `{impl_path}` and `{trait_path}`
- Discover implementation files:
  - `*ServiceImpl.java` in `{impl_path}` — Adapter layer (partially generated)
  - `*Repository.java` in `{trait_path}` — Repository trait extensions
  - `*DomainService.java` in `{impl_path}` — Domain logic
  - `*Controller.java` in `{generated_path}` — Generated controllers (read-only reference)
- If zero implementation files found, HALT: "No implementation files found in target path"

Present summary: **N** files found (ServiceImpl: X, Repository: Y, DomainService: Z). HALT and wait for user confirmation to proceed.

### Step 2: Load Rules

Load the golden rule file (primary) and conditional deep rules:
1. `{apihug_impl_golden_rule}` — **ALWAYS** load (core architecture/security/pattern rules)
2. `{apihug_impl_front_vben_guide}` - **OPTIONAL** load if front-end implement involved

If **focus** = `all`, load all five rule files.

### Step 3: Review — Architecture Layer Checks

For each implementation file, verify Onion Architecture compliance:

#### ServiceImpl Checks (Adapter Layer)
- [ ] Class is AUTO-GENERATED by wire — not manually created
- [ ] Class-level and method-level comments from generation are PRESERVED (not removed)
- [ ] Only method bodies are added/modified — no class structure changes
- [ ] No query logic — ALL queries delegated to Repository trait
- [ ] No direct database access (no EntityManager, JdbcTemplate, DataSource)
- [ ] No `builder.done()` calls — controller handles response
- [ ] Follows builder pattern: `builder.success()`, `builder.error()`, or throws `HopeErrorDetailException`
- [ ] No Lombok annotations — explicit getters/setters/constructors only

#### Repository Trait Checks (Infrastructure Layer)
- [ ] Class-level comments from generation are PRESERVED
- [ ] All query logic is HERE (not in ServiceImpl)
- [ ] Uses EasyCriteria or Spring Data Query Methods correctly
- [ ] No N+1 query patterns
- [ ] Pagination uses DB-level `findAll(criteria, pageRequest)` — no in-memory pagination
- [ ] Returns proper types: `Page<Entity>`, `List<Entity>`, `Optional<Entity>`

#### DomainService Checks (Domain Layer)
- [ ] Business logic is properly encapsulated
- [ ] No direct HTTP/controller concerns (no HttpServletRequest, ResponseEntity)
- [ ] Error handling uses `HopeErrorDetailException` (not RuntimeException)
- [ ] No RBAC code — zero `@PreAuthorize`, `@Secured`, `@RolesAllowed`

### Step 4: Review — Security Checks

- [ ] **ZERO RBAC code** in ServiceImpl/DomainService — proto authorization handles all RBAC
- [ ] No `@PreAuthorize`, `@Secured`, `@RolesAllowed` annotations anywhere in handwritten code
- [ ] No manual role/permission checks in method bodies
- [ ] Current user accessed via `HopeContextHolder.customer()` (or `customerWithContext()`) only
- [ ] No `SecurityContextHolder` direct access
- [ ] No manual JWT parsing or token validation

### Step 5: Review — Error Handling Checks

- [ ] Business errors use proto-defined error codes (not ad-hoc strings)
- [ ] Errors thrown as `HopeErrorDetailException` with proto error reference
- [ ] No raw `RuntimeException`, `IllegalArgumentException`, `IllegalStateException` for business errors
- [ ] ServiceImpl uses `builder.error()` or lets DomainService exceptions propagate
- [ ] No swallowed exceptions (empty catch blocks)
- [ ] No `e.printStackTrace()` — use proper logging

### Step 6: Review — Pagination Checks

- [ ] No `subList()`, `skip()`, `limit()` on full result sets — ALWAYS DB-level pagination
- [ ] PageRequest properly passed from ServiceImpl to Repository
- [ ] Page index conversion handled correctly (0-based internally)
- [ ] Repository returns `Page<Entity>` for paginated queries
- [ ] No manual page wrapping — framework wraps `Result<Pageable<T>>`

### Step 7: Review — Generated Code Boundary

- [ ] `src/generated/` files are NOT modified
- [ ] `src/main/generated/` files are NOT modified
- [ ] Controller files are read-only (generated)
- [ ] Entity files are read-only (generated)
- [ ] DSL/DAO files are read-only (generated)

### Step 8: Classify Findings

Classify each finding into one severity level:

- **CRITICAL** — Mandatory rule violation (any `NEVER`/`FORBIDDEN`/`ZERO TOLERANCE` rule). Architecture or security breach.
- **WARNING** — Best practice violation or pattern deviation. Should fix for correctness.
- **INFO** — Style suggestion or minor improvement opportunity.
