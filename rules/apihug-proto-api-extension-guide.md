# API Extension Guide

Map Proto RPCs to REST APIs with OpenAPI/Swagger generation.

- **Extension packages**: `apihug/protobuf/swagger/swagger.proto` + `annotations.proto`
- **Scope**: Service / Method (rpc) / Message / Field
- **Source**: `{project-root}/_bmad/protobuf/` (swagger, mock, extend, domain)


## MANDATORY EXECUTION RULES (READ FIRST):

- ­¤øæ **NEVER** add `page`, `size`, `sort` to Request Message ŌĆö framework injects via `pageable: true`
- ­¤øæ **NEVER** wrap response in `PageableXxx` or `XxxList` ŌĆö framework wraps `Result<Pageable<T>>`
- ­¤øæ **NEVER** use `string` for enum values ŌĆö use `EnumType` with full package path
- ­¤øæ **NEVER** add file-level options (`java_package`, `go_package`) ŌĆö build system injects
- ­¤øæ **NEVER** import `domain/` entities in `api/` layer ŌĆö API layer must NOT depend on Domain
- ­¤øæ **NEVER** use SQL reserved words as column names ŌĆö use prefixes: `user_name`, `order_no`
- ­¤øæ **NEVER** modify `_full.xml` migration file ŌĆö auto-generated by wire
- ­¤øæ **NEVER** add redundant "ENUM" suffix ŌĆö `STATUS_ACTIVE` Ō£ģ, `STATUS_ENUM_ACTIVE` ŌØī
- ­¤øæ **NEVER** use relative type references or import ŌĆö always use full package reference and full path import, `com.example.Order order =1` Ō£ģ, `Order order = 1` ŌØī
- ­¤øæ **NEVER** create Request Message with only 1-2 fields ŌĆö use `google.protobuf.Empty` + `parameters{}`
- ­¤øæ **NEVER** create proto message with single repeated field to support List, use `input_repeated` and `output_repeated` in method option.
- ­¤øæ **NEVER** use `DTO` when define request response message, ANY definition MUST have real business meaning!
- ­¤øæ **NEVER** put all code in `api`, `domain`, `infra`, Each feature should create individual sub dir in those main dir!
- ­¤øæ **NEVER** define INNER message type
- ­¤øæ **NEVER** add File-level ANY options, Enums are **FIRST-CLASS types**, NOT string conventions.
- ­¤øæ **NEVER** use `string` with enum values in comments
- Ō£ģ **ALWAYS** run `gradlew :{module}:wire` after proto changes ŌĆö sync generated code
- Ō£ģ **ALWAYS** use `snake_case` for proto fields ŌĆö convention compliance
- Ō£ģ **ALWAYS** define enums in `infra/{feature}/constant.proto`(or `error.proto`) ŌĆö centralized, shared
- Ō£ģ **ALWAYS** use full package path: `com.example.user.infra.UserStatusEnum`
- Ō£ģ **ALWAYS** add table prefix: `SYS_`, `BIZ_`, `CFG_`, `LOG_` ŌĆö avoid reserved words
- Ō£ģ **ALWAYS** use `pageable: true` for pagination ŌĆö single source of config
- Ō£ģ **ALWAYS** define errors with `code`, `http_status`, `phase`, `severity`
- Ō£ģ **ALWAYS** use enum in both `api` and `domain` layers ŌĆö same type flows through all layers
- Ō£ģ **ALWAYS** try best to fulfill all the proto extension meta, make sure they have business meaning!

### Proto Package Structure ŌĆö Module-First

Every feature should have a sub dir, NEVER MESS them together with others!


## 4. Operation Field Details

### 4.1 Basic Description

#### `description` (string)

- **Purpose**: Detailed description of the endpoint
- **Example**: `description: "Upload file metadata"`

#### `summary` (string)

- **Purpose**: Short summary (<120 chars) shown in Swagger UI
- **Example**: `summary: "Create an order"`

#### `tags` (repeated string)

- **Purpose**: Group endpoints by business module
- **Example**:
```proto
tags: "user";
tags: "order";
```

### 4.2 Request/Response Configuration

#### `request_name` (string)

- **Purpose**: Name for the request body parameter (code generation)
- **Example**: `request_name: "petRequest"`

#### `pageable` (optional bool)

**Type**: `optional bool` ** (triŌĆæstate boolean) **  
**Purpose**: Enable pagination

**Effects**:
- Request: automatically injects `page`, `size`, `sort` parameters ŌĆö framework-managed
- Response: wrapped in a `Page<T>` structure

> ŌØī **Never** add `page`, `size`, or `sort` fields to the Request Message manually. Setting `pageable: true` is the only correct way to enable pagination.

**Example**:
```proto
pageable: true
```

#### `input_repeated` / `output_repeated` (optional bool)

**Type**: `optional bool`  
**Purpose**: Mark request/response as a list

**Example**:
```proto
input_repeated: true;   // Request is List<T>
output_repeated: true;  // Response is List<T>
```

#### `body_empty` (bool)

**Purpose**: Allow empty response body (e.g., POST returning nothing)

**Example**:
```proto
body_empty: true
```

### 4.3 Media Types

#### `consumes` (repeated string)

- **Purpose**: Accepted ContentŌĆæTypes
- **Common values**: `"application/json"`, `"multipart/form-data"`
- **Example**:
```proto
consumes: "multipart/form-data";
```

#### `produces` (repeated string)

- **Purpose**: Returned ContentŌĆæTypes
- **Example**:
```proto
produces: "application/json";
```

#### `response_media_type` (MediaType enum)

**Type**: `Operation.MediaType` enum  
**Purpose**: Specify response media type

**Common values**:

| Enum Value            | MIME Type              | Use Case       |
|-----------------------|------------------------|----------------|
| `APPLICATION_JSON`    | application/json       | JSON (default) |
| `TEXT_PLAIN`          | text/plain             | Plain text     |
| `TEXT_HTML`           | text/html              | HTML page      |
| `APPLICATION_PDF`     | application/pdf        | PDF file       |
| `APPLICATION_XLSX`    | application/vnd...xlsx | Excel file     |
| `IMAGE_PNG`           | image/png              | PNG image      |
| `MULTIPART_FORM_DATA` | multipart/form-data    | File upload    |

**Example**:
```proto

```

### 4.4 Parameter Definition ŌÜĀ’ĖÅ

#### Four Rules ŌĆö Memorise These

1. **PATH params** (`/{id}`) ŌĆö **always** in `parameters{}`, no exceptions.
2. **Query params with a Request Message** ŌĆö define as Message fields; never repeat in `parameters{}`.
3. **Only PATH params / Ōēż 3 simple query params / no body** ŌĆö use `google.protobuf.Empty` as request + `parameters{}`.
4. **Pagination** ŌĆö **never** add `page`/`size`/`sort` to the Request Message; set `pageable: true` in the operation option and the framework injects them automatically.

> ŌØī Never create a Request Message solely to hold PATH parameters ŌĆö they are already declared in `parameters{}`.
> ŌØī Never define pagination fields (`page`, `size`, `sort`) manually in a Request Message ŌĆö use `pageable: true` instead.

#### `parameters` field reference

| Field    | Type       | Description                                        |
|----------|------------|----------------------------------------------------|
| `name`   | string     | Parameter name                                     |
| `in`     | IN enum    | `QUERY` / `PATH` / `HEADER` / `COOKIE` / `SESSION` |
| `schema` | JSONSchema | Constraints and description                        |
| `plural` | bool       | Array parameter                                    |

#### Examples

**GET by ID ŌĆö only PATH param:**
```proto
// Ō£ģ Empty request + PATH param in parameters{}
rpc GetOrder (google.protobuf.Empty) returns (OrderDetail) {
  option (hope.swagger.operation) = {
    get: "/orders/{orderId}";
    parameters: {
      parameter: { name: "orderId"; in: PATH; schema: { format: INTEGER; } }
    }
  };
}

// ŌØī WRONG ŌĆö Message just for a PATH param is redundant
message GetOrderRequest { int64 order_id = 1; }
rpc GetOrder (GetOrderRequest) returns (OrderDetail);
```

**Simple query ŌĆö few params, no Message:**
```proto
// Ō£ģ Empty request + PATH + 1 query param in parameters{}
rpc GetUser (google.protobuf.Empty) returns (UserDetail) {
  option (hope.swagger.operation) = {
    get: "/users/{userId}";
    parameters: {
      parameter: { name: "userId";        in: PATH;  schema: { format: INTEGER; } };
      parameter: { name: "with-profile";  in: QUERY; schema: { format: BOOLEAN; } }
    }
  };
}
```

**Complex query ŌĆö define a Request Message:**
```proto
// Ō£ģ Many query params ŌåÆ Request Message; PATH param still in parameters{}
rpc SearchProducts (SearchProductsRequest) returns (ProductSummary) {
  option (hope.swagger.operation) = {
    get: "/tenants/{tenantId}/products";
    pageable: true;
    parameters: {
      parameter: { name: "tenantId"; in: PATH; schema: { format: INTEGER; } }
    }
    // query fields come from SearchProductsRequest ŌĆö NOT repeated here
  };
}

message SearchProductsRequest {
  string keyword   = 1 [(hope.swagger.field) = {description: "search keyword"}];
  int32  category  = 2 [(hope.swagger.field) = {description: "category id"}];
  int32  min_price = 3 [(hope.swagger.field) = {description: "min price"}];
  int32  max_price = 4 [(hope.swagger.field) = {description: "max price"}];
}
```

### 4.5 Authorization

More details refer the `apihug-impl-security-rbac-guide`

#### `authorization` (Authorization message)

**Method 1: LowŌĆæLevel Risk Mode**

```proto
authorization: {
  low_limit_risky_mode: ANONYMOUS   // or LOGIN / ACTIVE
}
```

**LowLimitRiskyMode enum**:

| Enum Value  | Meaning                           |
|-------------|-----------------------------------|
| `ANONYMOUS` | No login required                 |
| `LOGIN`     | Login required                    |
| `ACTIVE`    | Login required and account active |

**Method 2: RBAC (Roles + Permissions)**

```proto
authorization: {
  rbac: {
    roles: {
      roles: ["ROLE_ADMIN", "ROLE_USER"]
    };
    authorities: ["USER_CREATE", "USER_DELETE"];
    combinator: AND;   // or OR
  }
}
```

**RBAC fields**:
- `roles.roles`: List of role names
- `authorities`: List of permissions (must be defined in an Authority enum)
- `combinator`: AND (all required) or OR (any required)

## 6. FieldŌĆæLevel Extension (JSONSchema)

### 6.1 Basic Fields

#### `description` (string)

- **Purpose**: Field description
- **Example**: `description: "User ID"`

#### `example` (string)

- **Purpose**: Example value (for documentation and frontend)
- **Example**: `example: "13800138000"`

#### `title` (string)

- **Purpose**: Field title
- **Example**: `title: "User Name"`

#### `default` (string)

- **Purpose**: Default value (JSON Schema level)
- **Example**: `default: "ACTIVE"`

### 6.2 Emptiness Control (Choose One)

**Important**: The following three fields belong to a `oneof EmptyConstraint` ŌĆō only one may be set.

#### `empty` (bool)

- **Applies to**: All types
- **Meaning**:
    - Primitive types: not null
    - Strings: not empty string (may contain spaces)
    - Collections: size > 0
- **Example**:
```proto
empty: false   // must not be empty
```

#### `blank` (bool)

- **Applies to**: Strings only
- **Meaning**: Must not be blank (i.e., cannot consist only of spaces)
- **Example**:
```proto
blank: false   // must not be blank
```

#### `nullable` (bool)

- **Applies to**: NonŌĆæstring types
- **Meaning**: Must not be null
- **Example**:
```proto
nullable: false   // must not be null
```

**Selection guide**:
```
String, forbid empty string   ŌåÆ empty: false
String, forbid blank string   ŌåÆ blank: false
Number/date, forbid null      ŌåÆ nullable: false
Collection, forbid empty      ŌåÆ empty: false
```

### 6.3 Length Constraints

#### String length


#### Collection size

**Type**: `uint64`

```proto

```

#### Object property count

**Type**: `uint64`


#### Multiple of

**Type**: `double`

## 8. Key Type Distinctions

### 8.1 bool vs. string

ŌØī **Incorrect**:
```proto

```

Ō£ģ **Correct**:
```proto

```

### 8.2 uint32/uint64/int64/double vs. wrapper types

ŌØī **Incorrect** (deprecated syntax):
```proto

```

Ō£ģ **Correct**:

### 8.3 optional bool vs. bool

**proto3 new syntax**: `optional bool` for triŌĆæstate boolean.

```proto

```

**Difference**:
- `bool`: default false (two states)
- `optional bool`: may be unset (three states)
